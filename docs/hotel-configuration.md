# 酒店配置中心开发说明

## 1. 目标

酒店配置中心用于把原来写死在 demo 数据中的价格规则，升级为每家酒店可单独配置的业务参数。

第一版目标：

- 每家酒店可以设置基本信息。
- 每个房型可以设置基准价、最低价、最高价和周末加价。
- 系统可以选择是否用配置价覆盖当前价。
- 推荐价会受到最低价和最高价保护。
- 配置可以导出为 JSON，后续可导入复用。

这一步让产品从“演示工具”向“可定制交付工具”前进。

## 2. 当前实现

新增文件：

```text
src/hotel_config.py
```

主要功能：

- `DEFAULT_HOTEL_CONFIG`：默认酒店配置。
- `ensure_hotel_config()`：确保 Streamlit session 中有酒店配置。
- `render_hotel_configuration(...)`：渲染酒店配置页面。
- `apply_config_to_current_prices(...)`：根据配置生成当前价。
- `room_bounds_from_config(...)`：把每个房型的最低价、最高价传给推荐引擎。
- `config_to_json_bytes(...)`：导出 JSON 配置。
- `load_config_from_upload(...)`：上传并导入 JSON 配置。

## 4. 配置如何影响价格

### 4.1 当前价

如果开启：

```text
用配置价覆盖当前价
```

系统会根据：

```text
当前价 = 房型基准价 + 周末加价
```

再通过价格尾数规则处理。

例如：

```text
标准大床房基准价：388
周末加价：40
原始当前价：428
价格尾数规则：中国酒店吉祥尾数（6/8/9）
最终当前价：428
```

如果关闭该选项，系统会使用上传的 `current_prices.csv` 或数据文件中的 `current_price`。

### 4.2 推荐价

推荐引擎会先根据入住率、库存、pickup 和时间压力计算推荐价，然后应用：

```text
最低价保护
最高价保护
价格尾数规则
```

例如：

```text
系统原始推荐价：715
高级大床房最高价：688
最终推荐价：688
风险提示：已应用最高价保护
```

## 8. 淡旺季日历（Season Calendar）

### 8.1 功能说明

酒店配置中心新增"季节日历"区块，允许为任意日期区间设置需求倍率，系统在生成调价建议时会将季节信号纳入评分。

进入酒店配置页 → 展开"季节日历"区块，即可增删季节段。每一行包含：

| 字段 | 说明 | 约束 |
|---|---|---|
| 季节名称 | 在推荐原因中显示 | 最长 40 字符 |
| 开始日期 | ISO 格式（YYYY-MM-DD） | ≤ 结束日期 |
| 结束日期 | ISO 格式（YYYY-MM-DD） | ≥ 开始日期 |
| 需求倍率 | 旺季 > 1.0，淡季 < 1.0，正常 = 1.0 | 0.1 ~ 5.0 |

修改后点击"保存季节配置"生效，最多可配置 50 个季节段。

### 8.2 如何影响调价建议

倍率 **只影响评分和推荐原因**，不直接修改基准入住率传入收益模拟：

- 倍率 > 1.05（旺季）：评分 +1，推荐原因显示"入住日期在旺季（XX），需求预期高于历史均值"
- 倍率 < 0.95（淡季）：评分 -1，推荐原因显示"入住日期在淡季（XX），需求预期低于历史均值"
- 1.05 ≥ 倍率 ≥ 0.95：不影响评分（中性区间）

**设计决策**：即使是旺季，如果当期实际入住率远低于历史基准，系统仍会建议向下调价（不低于最低价），以便从竞品酒店争取客源。因此季节信号作为附加评分项，而非强制抬高模拟基线。

### 8.3 多区间重叠规则

同一入住日期匹配多个季节段时，取**最高倍率**（旺季优先）。例如"国庆黄金周"（2.0）和"秋季旺季"（1.3）重叠时，国庆日期显示旺季信号。

### 8.4 默认季节配置（演示用）

首次使用时，系统预置以下演示用中国节假日季节段，可根据实际酒店情况自行修改或清空：

| 名称 | 时间 | 倍率 |
|---|---|---|
| 春节黄金周 | 1月25日—2月8日 | 1.8 |
| 国庆黄金周 | 10月1日—10月7日 | 2.0 |
| 暑期旺季 | 7月1日—8月31日 | 1.3 |
| 五一小长假 | 5月1日—5月5日 | 1.5 |
| 冬季淡季 | 11月1日—11月30日 | 0.7 |

## 9. Pickup 速度基准

### 9.1 什么是 Pickup 速度基准

系统会自动从历史订单中，计算**同酒店、同房型、同周末/工作日**类型的 14 天预订速度中位数，作为基准。

- 当前 14 天 pickup ÷ 基准 14 天 pickup = 速度比率
- 比率 ≥ 130%：近期预订明显快于历史同期，评分 +1
- 比率 ≤ 70%：近期预订明显慢于历史同期，评分 −1
- 基准样本 < 3 个历史日期时，自动回退到绝对值判断（pickup_14d ≥ 4 加分、临近到期且 ≤ 1 减分）

### 9.2 使用前提

历史订单中需要有足够的 `stayed` / `confirmed` 状态记录：样本越多（尤其是涵盖同类型日期），基准越准确。全新酒店或历史数据极少时，速度比率不生效，不影响其他信号正常运行。
