# 营销与产品开发路线图

## 1. 产品定位

本项目不应定位为“完全自动替代销售总监的 AI 改价工具”，而应定位为：

```text
为中小酒店、精品酒店、民宿和酒店管理教学场景提供可解释、可审批、可发布的智能调价助手。
```

核心价值：

- 用历史订单、库存、当前价和 pickup 数据辅助定价。
- 把“拍脑袋定价”变成“数据辅助决策”。
- 保留销售总监、店长或收益经理的最终控制权。
- 通过审批、人工修改标记和审计日志增强可信度。
- 后续逐步连接 PMS / Channel Manager，实现真正的一键发布。

推荐中文名称：

```text
酒店收益管理与智能调价助手
```

推荐英文名称：

```text
ONE Hotel Pricing Assistant
ONE Revenue Assistant
Hotel Revenue & Pricing Assistant
```

## 2. 商业模式选择

### 2.1 不建议优先做 exe / dmg

桌面软件看起来简单，但对本项目并不最优：

- Windows / macOS 要分别打包和测试。
- 客户电脑环境复杂，排错困难。
- 更新和维护成本高。
- 多用户审批、日志、权限和定时任务不方便。
- 后续连接 PMS / Channel Manager 不适合放在单机桌面软件里。

### 2.2 推荐主路线：SaaS + 私有化部署 + 咨询交付

推荐采用三种交付方式：

1. **在线 SaaS 版**
   - 面向小酒店、民宿、教学实训和轻量客户。
   - 通过网页登录使用。
   - 按月或按年订阅收费。

2. **私有化部署版**
   - 面向数据敏感客户、集团客户、学校机房或国内客户。
   - 部署到客户自己的服务器、国内云、内网服务器或学校服务器。
   - 收一次性部署费和年度维护费。

3. **咨询项目版**
   - 面向早期推广。
   - 以“收益管理数字化诊断 + 数据分析 + 系统部署 + 培训”的方式收费。
   - 适合作为第一批真实客户的进入方式。

## 3. 目标客户

### 3.1 第一批优先客户

优先寻找以下客户：

- 中小型单体酒店。
- 精品酒店。
- 民宿运营公司。
- 酒店管理与数字化运营专业院校。
- 酒店数字化培训机构。
- 有 Excel 订单数据但没有专业 RMS 的酒店。

### 3.2 不优先服务的客户

早期不建议优先服务：

- 大型国际连锁酒店集团。
- 已经使用成熟 RMS 的高端酒店。
- 需要同时接入多个复杂 PMS / OTA 的客户。
- 一开始就要求完全自动改价且无人工审批的客户。

适合酒店管理与数字化运营专业：


可包含：

- 教师账号。
- 学生账号。
- 内置 demo 数据。
- 实训任务。
- 报告导出。

- 多语言界面。

### 4.3 私有化部署版

| 类型 | 内容 | 建议价格 |

```text

```

```text
5000–30000 元 / 项目
```

根据客户规模、数据复杂度、是否需要现场培训调整。

## 5. 真实酒店配置中心

当前 demo 价格是写在程序中的演示数据。真实产品必须支持每家酒店配置自己的价格体系。

需要新增页面：

```text
酒店配置中心
Hotel Configuration
```

### 5.1 酒店基本信息

- 酒店名称
- 城市
- 商圈
- 酒店星级 / 定位
- 房间总数
- 是否允许自动发布
- 默认语言
- 货币单位

### 5.2 房型配置

每个房型应配置：

- 房型名称
- 房型代码
- 基准价
- 最低价
- 最高价
- 是否参与推荐
- 默认库存
- 对应 PMS / Channel Manager 房型 ID

示例：

| 房型 | 基准价 | 最低价 | 最高价 |
|---|---:|---:|---:|
| 标准大床房 | 388 | 328 | 588 |
| 高级大床房 | 468 | 398 | 688 |
| 家庭房 | 588 | 488 | 888 |

### 5.3 价格规则配置

每家酒店应可配置：

- 周末加价规则
- 节假日加价规则
- 临近入住调价规则
- 提前预订折扣规则
- 单次最大涨跌幅
- 推荐周期
- 价格尾数规则，例如 6/8/9、按 5 取整、按 1 取整
- 最低价保护
- 最高价保护

### 5.4 审批设置

- 审批人
- 是否允许批量采纳
- 是否允许人工修改最终价
- 是否必须填写修改原因
- 是否允许自动发布
- 推送前是否必须二次确认

## 6. 定价算法路线

真实算法不应只根据 demo 规则计算，而应形成可解释的价格推荐框架：

```text
推荐价 = 基准价 × 需求系数 × 库存系数 × 时间系数 × 节假日系数 × 竞品系数
```

然后再应用：

```text
最低价 / 最高价限制
单次最大涨跌幅限制
价格尾数处理
人工审批
```

### 6.1 第一阶段：可解释收益模拟算法

当前算法保留可解释需求信号，但最终推荐价由候选价收益模拟选择：

- 入住率高于历史同类日期 → 建议涨价
- 入住率低于历史同类日期 → 建议降价
- 剩余库存少 → 建议涨价
- 临近入住且库存多 → 建议降价
- 14 天 pickup 强 → 建议涨价
- 14 天 pickup 弱 → 谨慎降价或保持
- 候选价收益模拟 → 在可执行价格集合中选择预期收益最高的价格

### 6.2 第二阶段：配置化算法

将规则参数变成酒店可配置项：

- 高入住率阈值
- 低入住率阈值
- 剩余库存阈值
- pickup 阈值
- 周末系数
- 节假日系数
- 最大涨跌幅

### 6.3 第三阶段：预测模型

在有足够真实数据之后，引入：

- 需求预测
- 取消率预测
- 价格弹性估计
- 竞品价格影响
- 活动和节假日影响

但不应过早引入黑箱模型。早期重点是可解释和可落地。

## 7. 发布与集成路线

不建议一开始直接接所有 PMS / OTA。应按以下顺序推进：

### 阶段 1：Excel / CSV 导出

- 当前已经支持 Excel 调价建议报表。
- 下一步可增加“发布模板导出”。
- 酒店可以人工导入或人工复制到后台。

### 阶段 2：Channel Manager 导入模板

为具体渠道管理系统生成固定格式文件：

- 日期
- 房型代码
- 价格计划代码
- 最终批准价
- 最小连住
- 可售状态
- 库存限制

### 阶段 3：接入一个真实 API

选择一个具体目标系统做深度接入，例如：

- Mews
- Cloudbeds
- SiteMinder
- Apaleo
- 国内某个 PMS / Channel Manager

实现：

- API token 配置
- 房型映射
- 价格计划映射
- dry-run 预检查
- 推送结果回写
- 错误重试
- 操作日志

### 阶段 4：多系统 adapter

形成统一接口：

```text
publish_prices(approved_prices, target_system_config)
```

每个 PMS / Channel Manager 单独写 adapter。



建议架构：

```text

```

早期不要过度工程化，优先让真实客户能试用。

## 10. 推广策略

### 10.1 演示话术

推荐话术：

```text
这不是全自动替代销售总监的黑箱系统，而是一个可解释、可审批、可发布的酒店调价助手。
系统先根据历史订单、入住率、库存和 pickup 给出建议价，销售总监可以采纳或手动修改，最后一键导出或发布，并保留完整审计记录。
```

### 10.2 第一次客户演示流程

1. 打开销售演示看板。
2. 展示入住率、ADR、RevPAR 和调价机会。
3. 解释推荐价为什么上调或下调。
4. 切换价格尾数规则。
5. 打开价格审批与发布。
6. 批量采纳部分推荐价。
7. 手动修改一两行最终价。
8. 展示黄色人工修改标记。
9. 模拟一键推送。
10. 下载审计日志。

### 10.3 第一批客户获取方式

优先尝试：

- 酒店管理学校实训合作。
- 中小酒店免费诊断。
- 民宿运营公司试用。
- 酒店数字化课程演示。
- 通过 LinkedIn / 微信 / 邮件联系酒店经理或学校老师。

### 10.4 早期成交方式

不要一开始推“订阅制软件”，而是先推：

```text
一次性收益管理数字化诊断 + 工具部署 + 培训
```

交付内容：

- 分析酒店历史订单数据。
- 设置房型和基准价。
- 部署调价助手。
- 培训店长或销售总监。
- 给出第一版调价策略报告。

## 11. 关键风险

### 11.1 数据安全

酒店可能不愿意上传真实订单和房价数据到境外服务器。

应对：

- 提供 demo 数据演示。
- 提供国内云部署。
- 提供私有化部署。
- 支持客户本地部署。
- 明确不把客户数据提交到 GitHub。

### 11.2 客户不相信算法

应对：

- 强调可解释规则。
- 显示推荐原因。
- 保留人工审批。
- 支持手动修改。
- 保留审计日志。

### 11.3 PMS 接入复杂

应对：

- 先做 Excel/CSV 导出。
- 再做单个系统 API。
- 最后做多系统 adapter。

### 11.4 价格建议不符合酒店习惯

应对：

- 支持每家酒店设置基准价。
- 支持最低价和最高价。
- 支持价格尾数规则。
- 支持人工覆盖。
- 后续支持酒店自定义规则。

## 13. 当前推荐路线

最终推荐路线：

```text
演示版 → 咨询项目 → 配置化 SaaS → 私有化部署 → PMS / Channel Manager 接入
```

当前阶段最重要的不是马上做复杂算法，而是把产品包装成一个可信的业务系统：

```text
酒店配置
调价建议
人工审批
一键发布
审计日志
```

这五个模块打通后，项目就可以开始对外推广和试收费。
