# 自动价格监控与调价频率设计

本文档回答一个核心问题：既然系统可以每天检查价格，为什么不把它做成自动化？

结论：应该做自动化，但要分清 **自动检查 / 自动计算 / 自动发布** 三个层级。

```text
自动检查数据变化 -> 自动重新计算推荐价 -> 根据风险等级决定自动发布还是人工审批
```

当前 MVP 已经具备“自动计算推荐价”的核心模型。下一阶段应该把它升级为“自动监控任务”，定时读取最新订单、库存和价格，再自动生成新的调价建议。

## 1. 为什么自动化很有价值

同一个入住日期的最佳价格不是固定的。随着入住日期接近，以下变量每天都会变化：

- 已售房数
- 剩余库存
- 最近 7 天 / 14 天 pickup
- 取消订单
- 竞争环境
- 渠道促销状态
- 节假日或活动日临近程度

因此，价格也应该随这些信号重新计算。

例如：

```text
6月1日看 7月15日：还早，历史基准更重要。
7月1日看 7月15日：销售进度和 pickup 更重要。
7月14日看 7月15日：剩余库存和临近入住压力最重要。
```

这就是本项目从“静态报表”变成“收益管理助手”的关键。

## 2. 三个自动化层级

### Level 1：自动检查

系统定时拉取或读取最新数据：

- 新订单
- 取消订单
- 当前库存
- 当前已发布价格
- 渠道价格状态

输出：

```text
数据是否变化？
是否有异常？
哪些日期 / 房型需要重新计算？
```

### Level 2：自动计算

只要数据发生变化，系统自动重新运行推荐模型：

```text
最新数据 -> 指标计算 -> 候选价收益模拟 -> 新推荐价
```

输出：

- 推荐价
- 预期收益变化
- 置信度
- 风险提示
- 是否需要人工审批

### Level 3：自动发布

自动发布必须更谨慎。建议只允许低风险场景自动发布，高风险场景进入人工审批。

```text
高置信度 + 小幅调价 + 无异常 + 在价格上下限内 -> 可自动发布
低置信度 / 大幅调价 / 临近入住 / 异常库存 -> 人工审批
```

## 3. 推荐调价频率

### 普通日期

```text
每天 1 次自动检查和自动计算
默认人工审批后发布
```

适合大部分平日和远期日期。

### 重点日期

```text
每天 2-4 次自动检查和自动计算
价格变动进入审批队列
```

适合：

- 节假日
- 会展
- 演唱会
- 周末
- 城市活动
- pickup 突然增强
- 剩余库存快速下降

### 临近入住日期

```text
每 2-4 小时检查一次
只允许非常小幅、低风险调整自动发布
```

适合：

- 入住前 0-3 天
- 库存紧张
- 突然取消较多
- 突然新增订单较多

## 4. 自动发布护栏

即使做自动化，也不应一开始完全放开。

建议规则：

| 场景 | 自动计算 | 自动发布 | 说明 |
|---|---|---|---|
| 高置信度，小幅涨价 | 是 | 可选 | 例如涨幅 ≤ 5% |
| 高置信度，小幅降价 | 是 | 可选 | 例如降幅 ≤ 5%，且库存压力明显 |
| 中置信度 | 是 | 否 | 进入人工审批 |
| 低置信度 | 是 | 否 | 只作为观察信号 |
| 价格触及最低价 / 最高价 | 是 | 否 | 人工确认 |
| 临近入住且大幅调价 | 是 | 否 | 风险高 |
| 数据质量异常 | 否 / 部分 | 否 | 先修数据 |
| 渠道折扣叠加后低于净收益底线 | 是 | 否 | 需要人工确认 |

## 5. 触发式重新计算

除了定时任务，还应支持事件触发：

```text
新订单进入 -> 重新计算受影响日期 / 房型
订单取消 -> 重新计算受影响日期 / 房型
库存调整 -> 重新计算受影响日期 / 房型
竞争价格大幅变化 -> 重新计算相关日期
节假日 / 活动日导入 -> 重新计算相关日期
```

早期可以先做批处理：每天定时跑一次。后续接入 PMS / Channel Manager API 后，再改成事件触发或更高频轮询。

## 6. 推荐任务调度方案

### 当前 VPS 轻量方案

可用 systemd timer 或 cron：

```text
每天 07:00：拉取数据并生成建议
每天 12:00：重点日期重新检查
每天 18:00：重点日期重新检查
```

### SaaS 版本方案

可使用任务队列：

```text
Scheduler -> Pricing Job -> Recommendation Snapshot -> Approval Queue -> Publish / Export
```

推荐保存每次运行快照：

- run_id
- started_at
- data_version
- affected_dates
- affected_room_types
- generated_recommendations
- auto_publish_candidates
- approval_required_candidates
- warnings

## 7. 与 Channel Pricing Rules 的关系

自动化价格计算不能只看基准挂牌价，还要考虑渠道规则。

例如系统批准基准价 468：

```text
官网直销：468
官网会员：445
携程促销：431
Booking Genius：421 - 佣金
```

如果渠道折扣叠加后净收益太低，那么即使基准价看起来合理，也可能不适合自动发布。

因此自动发布前必须检查：

```text
Approved Base Price -> Channel Pricing Rules -> Channel Net Revenue -> Publish Guardrails
```

## 8. 本项目下一步建议

建议按以下顺序实现：

1. Channel Pricing Rules 数据结构和计算函数。
2. 在审批页面增加“生成渠道价预览”。
3. 增加渠道价 CSV 导出。
4. 增加自动价格监控任务，只生成推荐，不自动发布。
5. 增加自动发布候选筛选，但默认仍需人工确认。
6. 最后才接真实 Channel Manager API。

这样既能展示自动化价值，又不会过早承担自动写价风险。
