# 产品需求文档

## 1. 背景

酒店房价受入住日期、提前期、剩余库存、历史需求、节假日、渠道、取消率和竞品价格等因素影响。人工调价依赖经验，容易出现反应慢、价格过高导致空房、价格过低导致收益损失等问题。

本项目希望建立一个定价引擎，根据数据生成可解释的价格建议，帮助酒店更及时地调整房价并提升预期收益。

## 2. 目标

第一版目标：

- 预测未来指定入住日期和房型的需求变化
- 评估不同价格下的预计成交量和预期收益
- 推荐一个或多个候选价格
- 解释推荐原因和风险
- 支持人工确认后再执行调价

长期目标：

- 支持多酒店、多房型、多渠道定价
- 引入竞品价格、当地活动、天气和节假日数据
- 支持 A/B 测试和收益回测
- 为后续自动调价提供可审计基础

## 3. 用户角色

### 酒店收益经理

负责每日查看房价建议，判断是否接受系统推荐。

核心需求：

- 快速知道哪些日期需要调价
- 理解推荐价背后的原因
- 比较当前价、推荐价和历史类似日期价格

### 酒店运营负责人

关注整体收益、入住率和渠道表现。

核心需求：

- 查看一段时间内的收益趋势
- 识别淡旺季和异常日期
- 判断定价策略是否有效

### 数据或技术维护者

负责导入数据、检查数据质量和维护模型。

核心需求：

- 明确数据字段和口径
- 能复现实验结果
- 能发现异常输入和模型失效场景

## 4. 核心使用场景

### 场景 1：每日调价建议

收益经理每天查看未来 7、14、30、60 天的房价建议。

系统输出：

- 入住日期
- 房型
- 当前价
- 推荐价
- 预计收益变化
- 建议动作：上调、下调、保持
- 推荐理由
- 风险提示

### 场景 2：节假日或活动前调价

当某个日期预订速度高于历史同期，系统提示可能存在高需求机会。

系统输出：

- 需求异常信号
- 当前库存消耗速度
- 推荐提价幅度
- 与历史类似日期对比

### 场景 3：临近入住日期库存压力

当入住日期临近但剩余库存较多，系统提示降价或促销风险。

系统输出：

- 剩余库存
- 历史临近预订转化
- 推荐降价区间
- 是否建议渠道促销

## 5. 功能需求

### 5.1 数据导入

MVP 需要支持从 CSV 或表格导入以下数据：

- 历史订单
- 房型库存
- 历史价格
- 取消记录
- 日历特征

### 5.2 数据检查

系统需要检查：

- 日期字段是否合法
- 价格是否为正数
- 入住日期是否晚于或等于预订日期
- 取消订单是否被正确标记
- 房型名称是否一致
- 同一日期和房型的库存是否缺失

### 5.3 指标计算

系统需要计算：

- 每日房费收入
- 入住率
- ADR
- RevPAR
- Pickup
- 取消率
- 提前期分布

### 5.4 需求预测

系统需要预测：

- 指定入住日期的最终售出间夜数
- 指定日期和房型在未来窗口内的新增预订量
- 取消风险对净需求的影响

### 5.5 价格推荐

系统需要：

- 生成候选价格区间
- 估计每个候选价格下的预计销量
- 计算每个候选价格的预期收益
- 推荐收益最高或风险调整后最优的价格
- 给出推荐理由

### 5.6 人工审核

MVP 不自动改价。系统只输出建议，人工决定是否执行。

推荐结果需要包含：

- 当前价
- 推荐价
- 建议动作
- 预计收益变化
- 置信度
- 主要驱动因素
- 异常提醒

## 6. 非功能需求

- 可解释：每条建议必须说明主要依据。
- 可复现：同一份输入数据应能得到同一份输出。
- 可回测：历史数据上可以模拟推荐策略表现。
- 可审计：推荐结果应保留输入快照、模型版本和生成时间。
- 渐进复杂：先实现清晰的基线方法，再引入复杂模型。

## 7. MVP 验收标准

- 能导入一份酒店历史订单样例数据。
- 能生成每日、房型级别的基础指标。
- 能对未来日期输出当前价、推荐价和建议动作。
- 能展示推荐价的主要原因。
- 能用历史数据做回测，比较推荐策略与实际历史价格的收益差异。
- 能明确标记数据不足或模型不确定的日期。

## 8. 成功指标

第一阶段验证指标：

- 回测 RevPAR 相比历史实际策略有提升
- 推荐价格方向与人工经验大体一致
- 高风险建议能够被识别并降级为人工复核
- 需求预测误差在可接受范围内

具体阈值需要在拿到真实数据后设定。
