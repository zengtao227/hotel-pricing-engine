# 价格审批与发布使用手册

## 1. 当前价从哪里来

在真实酒店场景中，`current_price` 应来自酒店当前正在销售的价格来源，例如：

- PMS / 酒店管理系统
- Channel Manager / 渠道管理系统
- CRS / 中央预订系统
- 官网价格系统
- 酒店手工维护的 Excel 价目表

需要注意：OTA、比价网站或搜索引擎上看到的价格，很多时候是打折后、含促销、含会员权益或动态优惠后的成交参考价，不一定等于酒店内部维护的门市标价或挂牌价。收益管理系统里使用的 `current_price` 更应该代表酒店准备对外发布和管理的当前可售价格。

在当前演示版本中，如果没有上传真实 CSV，系统会使用 `src/sample_data.py` 自动生成 demo 数据。新版 demo 数据定位为“中国二线城市四星级商务酒店”，基础挂牌价为：

- Standard Double：388
- Superior Double：468
- Family Room：588

周末和高需求日期会在基础价上增加一定上浮，再通过价格尾数规则转换成更自然的展示价格，例如 428、508、628 等。

如果 VPS 上已经存在旧版 `sample_data/*.csv`，应用会优先读取旧 CSV。要使用新版 demo 价格，可以删除 VPS 上的 `sample_data` 目录，或重新生成 demo 数据。

## 2. 价格尾数规则

酒店真实挂牌价格通常不会直接展示算法算出来的精确值。算法可能算出 463.42，但对客展示时更可能使用 468、488、458、465 等更自然的价格点。

当前系统支持三种价格尾数规则：

1. **中国酒店吉祥尾数（6/8/9）**
   - 适合中国酒店销售演示。
   - 推荐价会尽量靠近算法计算值，同时优先使用 6、8、9 结尾。
   - 例如：算法原始价 463 可能显示为 468；算法原始价 586 可能显示为 588 或 589。

2. **按 5 取整**
   - 适合欧美或通用演示。
   - 例如：463 → 465，487 → 485。

3. **按 1 取整**
   - 保留最细粒度整数价格。
   - 适合后续做严谨算法测试。

默认策略为“中国酒店吉祥尾数（6/8/9）”。

## 3. 推荐价、最终批准价和人工修改

系统会先生成：

- 当前价：酒店当前正在卖的价格。
- 系统推荐价：算法根据入住率、库存、pickup、历史基准和价格尾数规则生成的建议价格。
- 最终批准价：销售总监或店长审核后准备发布的价格。

在“价格审批与发布”页面中：

- 可以批量采纳系统推荐价。
- 可以逐行修改最终批准价。
- 如果最终批准价不等于系统推荐价，系统会自动标记为“人工修改”。
- 人工修改行会用黄色提示。
- 已推送行会用绿色提示。
- 已拒绝行会用红色提示。

## 4. 当前版本的一键推送

当前版本的一键推送是 **模拟推送**，不会真的修改 PMS、Channel Manager 或 OTA 后台。

模拟推送的目的：

- 让客户看到完整业务闭环。
- 测试价格审批流程是否合理。
- 生成可审计的操作日志。
- 为后续真实 API 接入做准备。

模拟推送只会处理：

- 已勾选“推送”的行。
- 审核状态为“已批准”的行。

推送后系统会记录：

- 操作时间
- 操作人
- 酒店
- 房型
- 入住日期
- 当前价
- 系统推荐价
- 最终批准价
- 是否人工修改
- 审核备注
- 目标系统
- 推送状态

日志现在会同时：

- 在页面中显示。
- 支持下载为 Excel。
- 持久化保存到 `data/audit_logs/price_approval_publishing_log.csv`。

`.gitignore` 已经忽略生成的日志 CSV，避免把真实客户操作记录提交到代码仓库。

## 5. 回测分析

系统新增“回测分析”页面，用历史已发生的入住日期模拟当时的推荐价，并用实际售出房间数做静态销量回测。

当前回测口径：

```text
基准收入 = 当前价 * 实际售出房间数
推荐价静态收入 = 推荐价 * 实际售出房间数
静态收益变化 = 推荐价静态收入 - 基准收入
```

注意：当前版本假设售出房间数不随价格变化，用于快速验证推荐方向和解释逻辑。后续可以升级为带价格弹性的需求模型。

## 6. 真实后台系统接入顺序

不建议一开始就直接接入所有 PMS 或 OTA。最现实的路线如下：

### 第一步：Excel / CSV 导出

系统先生成酒店可以人工导入的价格表。适合演示、咨询项目和早期客户试用。

### 第二步：Channel Manager 导入模板

为具体渠道管理系统生成固定格式 CSV，例如日期、房型、价格、最小连住、可售状态等字段。

### 第三步：接一个真实 API

优先选择一个目标系统做深度接入，例如：

- Mews
- Cloudbeds
- SiteMinder
- Apaleo
- 酒店自有 PMS
- 国内某个渠道管理系统

实现：

- API token 配置
- 房型映射
- 价格计划映射
- 日期范围映射
- dry-run 预检查
- 推送结果回写
- 错误重试

### 第四步：多系统适配

把不同 PMS / Channel Manager 抽象成统一接口：

```text
publish_prices(approved_prices, target_system_config)
```

每个系统单独写 adapter。

## 7. 推荐产品定位

这个模块建议叫：

```text
价格审批与发布
Price Approval & Publishing
```

不要一开始对客户说“全自动改价”。更容易被接受的说法是：

```text
系统生成建议，销售总监审核确认，一键发布，并保留完整审计记录。
```

这样既能体现自动化效率，又保留酒店管理层的控制权。

## 8. 当前版本适合如何演示

推荐演示路径：

1. 打开销售演示看板，看整体入住率、RevPAR、调价机会和风险。
2. 打开调价建议，解释系统为什么建议上调或下调。
3. 切换价格尾数规则，展示推荐价如何变成更自然的挂牌价。
4. 打开回测分析，说明系统推荐方向在历史数据上的静态收益表现。
5. 打开价格审批与发布。
6. 点击批量采纳需调价项。
7. 手动修改一两行最终批准价，展示黄色人工修改标记。
8. 点击一键模拟推送。
9. 展示推送日志，并下载 Excel 审计记录。
